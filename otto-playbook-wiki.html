Agent OTTO — Playbook Engine Wiki

The unified skill taxonomy, playbook chaining model, and enhancement spec for OTTO Alpha v2

What is this?

This wiki documents how OTTO's skills actually work — what types exist, how they chain together, and how playbooks orchestrate them into complete workflows. It's the missing context layer that makes the Alpha demo self-explanatory.

What OTTO Alpha v2 unifies

Instead of creating something new, we're wiring these layers together through a skill taxonomy that lets OTTO automatically chain the right skills in the right order for any task. The persona determines how you work. The taxonomy determines what gets chained. The execution pattern determines how it runs.

Skill Taxonomy

Every skill falls into one of five execution types. This classification drives how skills chain together in playbooks.

Atomic

Single-Action Skills

One input → one output. No state. No branching. The building blocks.

Chain

Sequential Multi-Step Skills

Ordered steps where each output feeds the next input. A pipeline.

Loop

Iterative Refinement Skills

Repeat until a quality gate passes. Self-correcting cycles.

Orchestrator

Routing Skills

Dispatch to other skills based on context. Don't do work themselves — direct traffic.

Utility

Global Helpers

Available in every chamber. Context gathering, brainstorming, git ops, environment setup.

Classification Matrix

TypeExamplesExecutionStateCan Parent?
Atomic Security Review, Database Designer, Copywriting, Cold Email, Frontend Design Single invocation Stateless No — leaf nodes only
Chain Writing Plans, TDD (Red→Green→Refactor), Finishing a Branch Sequential pipeline Pass-through Yes — calls atomics in sequence
Loop Executing Plans (batch loop), Continuous PR Loop, De-Sloppify, Subagent-Driven Dev Repeat until done Accumulated Yes — wraps chains + atomics
Orchestrator Agent OTTO, Loki Mode, Ralphinho RFC Pipeline, Dispatching Parallel Agents Route + delegate Coordination Yes — top-level only
Utility Brainstorming, Using Git Worktrees, Context7 MCP, Exa Search MCP On-demand Varies No — called by any parent

How Types Nest

The nesting rule: Orchestrators call Loops and Chains. Loops call Chains and Atomics. Chains call Atomics. Utilities can be called at any level. Nothing calls an Orchestrator except the user (or another orchestrator in a multi-agent setup). 
Orchestrator (OTTO / Ralphinho)
├── Loop (Subagent-Driven Development)
│   ├── Chain (Writing Plans → TDD → Finishing Branch)
│   │   ├── Atomic (Security Review)
│   │   ├── Atomic (Frontend Design)
│   │   └── Atomic (Database Designer)
│   └── Utility (Git Worktrees)
├── Loop (Executing Plans — batch of 3)
│   ├── Atomic (task 1)
│   ├── Atomic (task 2)
│   └── Atomic (task 3)
│   └── [human checkpoint → next batch]
└── Utility (Brainstorming — used before any loop)

Skill Chaining Model

How parent-child relationships between skills create playbooks

The Parent-Child Contract

Every skill chain follows a simple contract: the parent skill defines WHAT to do, and the child skill defines HOW to do it. The parent passes context down and receives results back up.

RelationshipParent Passes DownChild Returns UpExample
Orchestrator → LoopPhase context, persona mode, acceptance criteriaCompletion status, artifacts producedOTTO → Subagent-Driven Dev
Loop → ChainCurrent task, iteration count, accumulated stateTask result, pass/fail, next actionExecuting Plans → Writing Plans
Chain → AtomicStep input, constraints, quality barStep output (review, design, code)TDD → Security Review
Any → UtilityQuery / context needContext / environment / infoAny → Brainstorming

Chaining Patterns

Pattern 1: Linear Pipeline

Brainstorm → Research → Write Plan → Implement → Review → Deploy

Every step completes before the next begins. Simple, predictable. Best for small features.

Pattern 2: Fan-Out / Fan-In (Parallel)

Brainstorm → Agent A: Frontend | Agent B: Backend | Agent C: Tests → Merge + Review

Independent work units run concurrently. Merge point validates integration. Best for medium features with separable concerns.

Pattern 3: DAG with Dependencies (Ralphinho)

RFC Intake
├── Unit A (no deps) ─────────────────→ Validate A → Merge
├── Unit B (no deps) ─────────────────→ Validate B → Merge
├── Unit C (depends on A) ────────────→ Validate C → Merge
└── Unit D (depends on B + C) ────────→ Validate D → Merge
                                              ↓
                                     Final System Verification

Most sophisticated. Dependency-aware execution with merge queue, eviction recovery, and complexity tiers. Best for large features (10+ files changed).

Chamber × Type Affinity

Each chamber has natural affinities for certain skill types:

BuildChain LoopImplementation follows plans (chains) with iterative refinement (loops)
VerifyAtomic LoopReviews are atomic checks; the swarm itself is a loop of checks

Existing Playbook Architecture

What we already have — distributed across 5 skill collections

Layer Map

The playbook engine exists today as five layers that were built independently. Each layer is a skill or skill set that handles one aspect of orchestrated execution.

Layer 1: Plan-to-Execution Bridge (Superpowers)
executing-plans — Batch execution with human checkpoints (default 3 tasks per batch)
writing-plans — Generates bite-sized task lists with file paths, commands, commit messages
subagent-driven-development — Fresh subagent per task with two-stage review (spec + quality)
dispatching-parallel-agents — Concurrent agents for independent problem domains
Dependency chain: brainstorming → writing-plans → [executing-plans OR subagent-driven-dev] → finishing-a-development-branch
Layer 2: Execution Patterns (Autonomous Loops)
Six patterns ranked by complexity:
1. Sequential Pipeline — claude -p shell chains
2. NanoClaw REPL — persistent interactive sessions
3. Infinite Agentic Loop — 2-prompt orchestrator + N sub-agents
4. Continuous PR Loop — auto-PR + CI wait + auto-merge
5. De-Sloppify — separate cleanup pass after implementation
6. Ralphinho DAG — RFC decomposition → dependency-aware execution → merge queue
Layer 3: DAG Orchestration (Ralphinho RFC Pipeline)
7-stage pipeline: RFC intake → DAG decomposition → Unit assignment → Parallel implementation → Validation → Merge queue → System verification
Key innovations: Complexity tiers (trivial/small/medium/large), worktree isolation per unit, separate context windows for review (eliminates author bias), merge queue with eviction recovery
Layer 4: Persona Routing (Agent OTTO)
17 PI personas × 4 workflow chambers × 393 skills = behavioral routing layer
OTTO determines WHO does the work (persona), WHAT tools they use (skill loadout), and WHEN they transition (chamber flow)
Layer 5: Platform Spec (pack.json)
Specifies but does NOT yet implement:
playbook-execute.yaml — DAG-based playbook workflows with step tracking and conditional branching
workflow-engine-spec.md — DAG builder, node catalog, execution runtime
PlaybookDAG.tsx — Visual DAG canvas for playbook workflow execution
DAGNode.tsx — Individual node in DAG workflow visualization
WorkflowBuilder.tsx — Visual workflow DAG builder with node palette and canvas

The Gap

These layers work independently but aren't wired together. Today, you have to manually know which execution pattern to use, manually load the right skills, and manually transition between phases. OTTO's job is to make this automatic.

Enhancement Spec

How to unify the existing distributed engine — no new systems, just wiring

Goal

Make OTTO automatically select the right execution pattern, chain the right skills, and manage phase transitions — using only the skills that already exist.

Three Enhancements

Enhancement 1: Skill Type Metadata
Add a type field to every skill's YAML frontmatter: atomic | chain | loop | orchestrator | utility
Add a children field listing skills this skill can invoke.
Add a chamber_affinity field: [discovery, build, verify, ship]

Example:
name: subagent-driven-development
type: loop
children: [writing-plans, finishing-a-development-branch, security-review, tdd-guide]
chamber_affinity: [build, verify]
requires: [using-git-worktrees]
This lets OTTO's routing engine understand skill relationships without hardcoding them.
Enhancement 2: Playbook Templates in OTTO
Add a references/playbooks.md to the agent-otto skill with canonical playbook templates for common tasks.

Each playbook template specifies: chamber flow, persona per phase, execution pattern to use, skill chain, and acceptance criteria.

OTTO matches the user's task description against playbook templates (same keyword scoring as Auto-Suggest) and loads the best match.
Enhancement 3: Execution Pattern Auto-Selection
OTTO picks the execution pattern based on task complexity:

Complexity SignalPatternSkill
1-3 files, single domainLinear Pipelineexecuting-plans (batch of 3)
4-10 files, 2+ domainsFan-Out / Fan-Indispatching-parallel-agents
10+ files, dependenciesRalphinho DAGralphinho-rfc-pipeline
Multi-day / continuousPR Loopcontinuous-pr-loop

The user doesn't need to know which pattern — OTTO detects complexity from the task description and plan output.

Implementation Priority

Phase 1 (now): Add type metadata to the ~30 most important skills. Create playbooks.md with 5 canonical templates.
Phase 2 (next): Build the auto-selection logic into OTTO's SKILL.md. Wire it into the HTML alpha demo.
Phase 3 (later): Implement pack.json specs (playbook-execute.yaml, PlaybookDAG.tsx) for the actual Airlock app.

Relay Agent Pattern

A passive agent that caches MCP server state so other agents don't need direct connections

The Problem

Today, every agent that needs data from an MCP server has to connect directly, authenticate, and query. This means:

• Every agent needs credentials for every MCP server it might use
• Redundant queries hit the same endpoints repeatedly
• Agent startup is slow because it has to discover what's available
• No shared state between agents working on the same constellation

The Relay Pattern

┌─────────────────────────────────────────────────┐
│  Relay Agent (always running, passive)           │
│                                                  │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐        │
│  │ GitHub   │ │ Supabase │ │ Linear   │ ...    │
│  │ Cache    │ │ Cache    │ │ Cache    │        │
│  └──────────┘ └──────────┘ └──────────┘        │
│       ↕              ↕            ↕             │
│  [MCP Server]   [MCP Server]  [MCP Server]     │
│                                                  │
│  State: last_sync, cache_ttl, change_log        │
└──────────────────┬──────────────────────────────┘
                   │ query/response
     ┌─────────────┼─────────────┐
     ↓             ↓             ↓
┌─────────┐  ┌─────────┐  ┌─────────┐
│ Agent A │  │ Agent B │  │ Agent C │
│(Scholar)│  │(Maverick│  │(Analyzer│
│         │  │         │  │         │
└─────────┘  └─────────┘  └─────────┘

How It Works

StepWhat happensWho
1. BootRelay connects to all registered MCP servers, pulls initial stateRelay Agent
2. CacheStores responses indexed by server + query type with TTLRelay Agent
3. QueryWorking agent asks relay: "what's in the GitHub repo?"Any Agent
4. RespondIf cache fresh → return cached. If stale → refresh, cache, return.Relay Agent
5. WatchRelay monitors for changes (webhooks or polling) and updates cacheRelay Agent

Implementation

The relay agent would be a skill that runs as a background process in the Claude Code terminal. It exposes a simple query interface that other skills call instead of connecting to MCP servers directly.

Key insight: The relay doesn't replace MCP servers — it sits between agents and servers as a cache + change-detection layer. Agents ask the relay first. If it has a fresh answer, the MCP server never gets hit. If there's been an update, only the relay refreshes.

This is essentially a read-through cache with change notifications. The relay skill would need: a cache store (in-memory dict or SQLite), TTL per server type, a change log for audit, and a query API that other skills can call.

Canonical Playbook: Landing Page

"I'm building the landing page for my website" — here's the complete skill chain from intent to deploy

This is the proof-of-concept. When a user says "build me a landing page," OTTO should automatically generate this entire playbook — selecting the persona, loading the skills, choosing the execution pattern, and managing transitions between phases.

The Full Playbook

1
Discovery Chamber — Scholar Lead

Intent Capture + Research

Understand what the user actually wants. What's the business? Who's the audience? What's the tone? What pages exist already? What's the tech stack?

Brainstorming (utility) Exa Search MCP (atomic) Context7 MCP (atomic) Technology Stack Evaluator (atomic)

Output: Design brief with audience, goals, tech constraints, content requirements

2
Discovery Chamber — Strategist Support

Architecture + Design Doc

Create the UX/UI design document. Define layout, sections, components, animations, responsive breakpoints. Present for user approval before any code is written.

UI/UX Pro Max (chain) Senior Architect (atomic) Frontend Design (atomic) Figma MCP (atomic)

Output: Design doc with wireframe, component list, color palette, typography, content map

3
Build Chamber — Maverick Lead

Implementation Plan

Break the design doc into bite-sized implementation tasks. Each task: files to create/modify, step-by-step code, commands, commit messages. This is where Writing Plans skill generates the task list.

Writing Plans (chain) SaaS Scaffolder (atomic)

Output: 10-20 implementation tasks with full instructions

4
Build Chamber — Maverick + Venturer

Autonomous Coding

Execute the plan. For a landing page (medium complexity, 4-10 files), use Fan-Out pattern: one agent handles layout/structure, another handles content/copy, another handles animations/interactions. They merge at the end.

Subagent-Driven Dev (loop) Dispatching Parallel Agents (orchestrator) Frontend Design (atomic) Web Artifacts Builder (atomic) Copywriting (atomic) Algorithmic Art (atomic) Using Git Worktrees (utility)

Output: Working landing page code across all component files

5
Verify Chamber — Analyzer Lead

Review Swarm

The 10+ skill review swarm runs. Security checks, performance profiling, accessibility audit, code quality review, cross-browser validation. Each review is an atomic skill. The swarm itself is a loop.

Security Review (atomic) Performance Profiler (atomic) Coding Standards (atomic) Senior QA Engineer (atomic) E2E Testing Patterns (atomic) Playwright MCP (atomic) De-Sloppify (loop)

Output: Review report with pass/fail per check, fix list

6
Verify Chamber — Specialist Support

Fix + Re-test

Address review findings. Re-run failing checks. Iterate until all gates pass. This is the De-Sloppify pattern — a separate cleanup pass that fixes quality issues from the implementation phase.

TDD Guide (chain) De-Sloppify (loop) Finishing a Branch (chain)

Output: Clean, passing codebase ready for deploy

7
Ship Chamber — Guardian Lead

Deploy + Monitor

Push to production. Set up monitoring. Create rollback plan. Validate in production environment.

CI/CD Pipeline Builder (chain) Vercel MCP (atomic) Cloudflare MCP (atomic) Checkly MCP (atomic) Runbook Generator (atomic)

Output: Live landing page with monitoring and rollback plan

8
Ship Chamber — Adapter Support

Handoff + Report

Tell the user what was built, what decisions were made, what to watch for. Generate the deployment report. This is where OTTO closes the loop.

Board Deck Builder (atomic) Quality Documentation Manager (atomic)

Output: Deployment report, architecture notes, monitoring dashboard link

DAG Execution (Ralphinho Integration)

For complex playbooks (10+ files), OTTO delegates to the Ralphinho RFC Pipeline as its execution engine. The flow becomes:

OTTO receives task
  → OTTO selects Ralphinho pattern (complexity detected)
  → Ralphinho decomposes into work units (DAG)
  → Each work unit gets a persona-appropriate agent:
     - Frontend units → Maverick agent
     - Backend units → Venturer agent
     - Test units → Specialist agent
     - Security units → Analyzer agent
  → Merge queue handles integration
  → OTTO runs final verification (Analyzer swarm)
  → OTTO manages deployment (Guardian)
The key innovation: OTTO provides the persona routing layer on top of Ralphinho's DAG execution. Ralphinho handles the mechanics (decomposition, dependencies, merge queue). OTTO handles the intelligence (which persona, which skills, which quality bar). Together they form the complete playbook engine.
